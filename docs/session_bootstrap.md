# Genealogy Research Assistant (GRA) — Session Bootstrap

*Version 1.0 — May 2026*
*Audience: All roles. This document defines the two session types supported by GRA and specifies what Python must provide and what Claude must do at the start of each session. Read `conceptual_model.md`, `data_dictionary.md`, and `reconstruction_algorithms.md` first.*

---

## 1. Overview

### GRA as a knowledge base platform

GRA is a knowledge base platform. Its purpose is to accumulate accurate, evidence-backed conclusions about people, places, relationships, and events from historical records. It does not answer research questions directly — that is the responsibility of consumer applications built on top of it.

The two operations that keep the knowledge base current are:

- **Ingest** — loading a CSV of source records into the evidence layer
- **Update Knowledge** — evaluating proposed linkages between new evidence and existing conclusions, and committing the decisions that update the knowledge base

These are the only two session types. Both are geographically scoped. Neither requires a research question.

### Typical workflow

A researcher building a knowledge base for a target community will typically follow this sequence:

1. Obtain a CSV for a source covering the target area (census DED, civil registration district, valuation townland set)
2. Run an ingest session to load the evidence layer
3. Run an update knowledge session to process the proposals generated by that ingest
4. Repeat for additional sources, layering evidence against the growing conclusion layer

Census returns are the natural starting point. They provide the richest initial snapshot of a community — households, ages, roles, occupations — and the household structure inference produces an immediate seed of person and relationship conclusions that all subsequent sources are evaluated against.

### Context discipline

Claude has no persistent memory between sessions. Everything Claude needs to reason correctly must be provided in the session packet. The packet is intentionally small — geographically scoped to the target area, bounded to the current proposal batch, and limited to the conclusions directly relevant to the proposals at hand.

Claude does not need the full documentation set loaded at runtime. Claude carries schema and domain knowledge. Python provides research state.

## 2. Ingest Session

### 2.1 Purpose

Load a CSV of source records into the evidence layer. This is a Python-driven operation. Claude's role is narrow: resolving parse exceptions that Python cannot handle automatically.

### 2.2 What Python does

1. Validate the CSV structure against `Source.column_schema`
2. For each row, construct a `Record` with `raw_text` set to the verbatim CSV row
3. Parse each row into a `RecordedEvent` and one or more `RecordedPersons` using the column schema
4. Apply R36 (date format) and R37 (record_parameters key match) validation
5. Commit all well-formed rows to the database
6. Collect parse exceptions — rows that could not be committed due to ambiguous or malformed field values
7. Produce an ingest summary and, if any exceptions exist, a parse exception report

### 2.3 What Claude does

Claude is only involved if Python produces parse exceptions. For each exception, Python provides:

- The raw CSV row verbatim
- The column schema for the source
- The specific field(s) that could not be parsed and the reason

Claude resolves the exception by providing the corrected field value(s) with a brief note explaining the interpretation. Claude does not modify `raw_text` — that field is sacrosanct. Claude only provides the parsed field values that Python will use for the `RecordedEvent` or `RecordedPerson` row.

**Exception types Claude can resolve:**

| Exception type | Example | Claude action |
|---|---|---|
| Ambiguous age | "abt 30", "inf", "—" | Provide normalised `age` integer or null |
| Non-ISO date | "10th Jany 1890", "Feby 1901" | Provide normalised ISO 8601 date |
| Unknown role abbreviation | "Hd", "Sr", "Wid" | Map to controlled vocabulary role |
| Garbled name | "Jn°", "Mich¹" | Expand to normalised forename |
| Missing required field | Row has no townland | Flag as unresolvable; Python stores with null |

**What Claude does not do in an ingest session:**

- Claude does not make linkage decisions during ingest
- Claude does not create or modify conclusion-layer objects
- Claude does not assess whether a record is about a known person

### 2.4 Session packet for ingest exceptions

```json
{
  "session_type": "ingest",
  "source": {
    "source_id": 3,
    "title": "Census 1901",
    "type": "census",
    "column_schema": ["document_id", "household_id", "line_number", "surname", ...]
  },
  "ingest_summary": {
    "rows_attempted": 312,
    "rows_committed": 308,
    "exceptions": 4
  },
  "parse_exceptions": [
    {
      "row_number": 47,
      "raw_text": "...",
      "column_schema": [...],
      "problem_fields": ["age_as_recorded"],
      "problem_detail": "Value 'inf' could not be parsed as integer age"
    }
  ]
}
```

### 2.5 Claude's response format for exceptions

For each exception, Claude returns a structured resolution:

```json
{
  "row_number": 47,
  "resolved_fields": {
    "age": null,
    "age_as_recorded": "inf"
  },
  "note": "Value 'inf' indicates infant. Age set to null; age_as_recorded preserved verbatim."
}
```

If Claude cannot resolve an exception, it returns `"resolved_fields": null` with a note explaining why.

### 2.6 Ingest session close

After exceptions are resolved and committed, Python runs the reconstruction pipeline against the newly ingested records:

1. Place resolution — normalise and score place strings against existing Place conclusions
2. Splink scoring — generate and score person linkage candidates
3. Apply genealogical constraints — adjust scores per GC01–GC22
4. Auto-commit linkages at or above ceiling (default 0.85) with `verified = 0`
5. Suppress candidates below floor (default 0.3)
6. Queue remaining proposals (0.3–0.85) for the update knowledge session

The ingest session is complete when the proposal queue is ready. Python then initiates an update knowledge session.

## 3. Update Knowledge Session

### 3.1 Purpose

Review the proposal queue generated by the most recent ingest and commit decisions that update the knowledge base. This is the primary collaborative session between Python and Claude.

Python handles everything that can be automated. Claude handles what requires domain reasoning — cases where the scoring algorithm has useful signal but not enough confidence to commit without interpretation.

### 3.2 What Python has already done before Claude sees anything

By the time a proposal reaches Claude, Python has:

- Ingested and validated the evidence layer records
- Run Splink probabilistic scoring across all candidate pairs
- Applied genealogical constraint penalty multipliers (GC01–GC22)
- Auto-committed high-confidence linkages (score ≥ 0.85)
- Suppressed below-floor candidates (score < 0.3)
- Ranked remaining proposals by score descending

Claude is not reviewing routine linkages. Claude is reviewing the cases automation could not resolve confidently.

### 3.3 What Claude does

For each proposal, Claude:

1. Reads the proposal features (name similarity, birth year estimate, place match, co-occurring persons, role)
2. Reads the candidate conclusion (the Person or Place it is proposed to match)
3. Applies domain reasoning beyond what Splink scores capture
4. Returns a decision: **accept**, **reject**, or **flag**

Domain reasoning Claude applies:

**Patronymic disambiguation** — same name, same townland, birth year arithmetic suggests generational namesake rather than same person. Claude cross-references the existing relationship graph (e.g. if a Patrick Mulligan b. ~1820 already has a concluded child John b. ~1848, a new record with "Patrick Mulligan, age 8" in 1901 is almost certainly a third-generation Patrick, not the original).

**Cross-source age conflict** — census age implies birth year X, civil registration implies birth year Y, difference exceeds tolerance. Claude assesses whether the conflict is within normal informant error or genuinely disqualifying.

**Name variant reasoning** — a name pair not in the `name_variant` table but plausibly equivalent given Irish linguistic context. Claude can accept the linkage and recommend a variant table addition.

**Co-occurring person coherence** — the proposed person match has a known spouse Mary and the new record contains a co-occurring Mary in a spouse role. Claude weights this as corroborating even if the name similarity score for the principal is borderline.

**Place string ambiguity** — a place string that normalises to something close to two different known Place conclusions. Claude assesses which is more geographically plausible given the source and other records in the batch.

**Relationship inference from role pairs** — a record contains a father/principal pair that would generate a parent_child relationship assertion. If a relationship of that type already exists between the concluded persons (from a previous source), this is corroborating evidence, not a duplicate.

### 3.4 Decision types

| Decision | Meaning | Python action |
|---|---|---|
| `accept` | Linkage is correct | Commit junction row with Claude's assessed score; `verified = 0` |
| `accept_verify` | Linkage is correct; researcher should review | Commit junction row; `verified = 0`; add to researcher review queue |
| `reject` | Linkage is incorrect | Suppress candidate; do not commit |
| `flag` | Insufficient information to decide | Add to flags table with Claude's reasoning; surface to researcher |
| `new_person` | Record does not match any existing Person | Create new Person conclusion; commit linkage |
| `new_place` | Place string does not match any existing Place | Create new Place conclusion; commit linkage |
| `variant_add` | Accept linkage; name pair should be added to variant table | Commit linkage; add variant table entry |

### 3.5 Session packet for update knowledge

Python provides a session packet at session open. The packet is geographically scoped and bounded to the current proposal batch.

```json
{
  "session_type": "update_knowledge",
  "scope": {
    "source_id": 3,
    "source_title": "Census 1901",
    "source_type": "census",
    "geographic_scope": "Tullynaught DED, Co. Donegal"
  },
  "community_summary": {
    "persons_concluded": 142,
    "relationships_concluded": 89,
    "places_concluded": 7,
    "records_in_scope": 312,
    "records_linked": 271,
    "records_unlinked": 41
  },
  "proposals": [
    {
      "proposal_id": "p_001",
      "record_id": 1047,
      "candidate_type": "person",
      "candidate_id": 23,
      "score": 0.71,
      "score_version": "v1.0",
      "features": {
        "surname_similarity": 0.94,
        "forename_similarity": 1.0,
        "birth_year_delta": 3,
        "place_match": true,
        "occupation_similarity": 0.82,
        "co_person_overlap": 0.67
      },
      "record_summary": {
        "name_as_recorded": "Patrick Mulligan",
        "age_as_recorded": "49",
        "birth_year_est": 1852,
        "role": "head",
        "place_as_recorded": "Tullynaught",
        "occupation_as_recorded": "farmer",
        "co_persons": ["Mary Mulligan (spouse, 44)", "John Mulligan (son, 22)", "Brigid Mulligan (dau, 19)"]
      },
      "candidate_summary": {
        "person_id": 23,
        "label": "Patrick Mulligan (Tullynaught ~1852)",
        "gender": "male",
        "names": ["Patrick Mulligan"],
        "birth_year_est": 1849,
        "supporting_records": 2,
        "known_relationships": ["couple → Mary (née Brennan)", "parent_child → John Mulligan b.~1879"]
      },
      "constraint_notes": [
        "GC03: census age drift within tolerance (delta 3, tolerance ±3)",
        "GC18: co-occurring Mary consistent with known spouse"
      ]
    }
  ],
  "active_flags": []
}
```

The `community_summary` gives Claude orientation without loading the full conclusion set. The `candidate_summary` contains only the fields directly relevant to the linkage decision — not a full Person object dump.

### 3.6 Proposal batch size

A single update knowledge session should contain no more than **20 proposals**. This keeps the session packet small, maintains Claude's reasoning quality across the batch, and produces a clean commit unit.

If the proposal queue contains more than 20 items, Python splits them into sequential sessions. Proposals are ordered by score descending within each batch — higher-confidence proposals first.

### 3.7 Claude's response format

Claude returns a decision array — one entry per proposal in the batch:

```json
{
  "session_type": "update_knowledge",
  "decisions": [
    {
      "proposal_id": "p_001",
      "decision": "accept",
      "assessed_score": 0.78,
      "reasoning": "Name and place match strong. Birth year delta of 3 years is within census age drift tolerance. Co-occurring Mary Mulligan is consistent with the known spouse Mary (née Brennan). Co-occurring John Mulligan son age 22 is consistent with the known parent_child relationship to John b.~1879. The birth year discrepancy (record implies 1852, existing conclusion 1849) is within informant error range. Linkage is sound.",
      "relationship_inferences": [],
      "variant_additions": []
    },
    {
      "proposal_id": "p_002",
      "decision": "new_person",
      "assessed_score": null,
      "reasoning": "No existing Person conclusion matches this record. Name, place, and birth year combination is distinct from all candidates above floor. Creating new Person conclusion.",
      "suggested_label": "Brigid Mulligan (Tullynaught ~1882)",
      "suggested_gender": "female",
      "relationship_inferences": [
        {
          "type": "parent_child",
          "person_id_1": 23,
          "person_id_2": "new",
          "basis": "role pair: head + child in census record",
          "prior_score": 0.85
        }
      ],
      "variant_additions": []
    }
  ],
  "session_notes": "Batch processed cleanly. No conflicts detected. One new person created (Brigid Mulligan). Recommend adding 'Brigt' as abbreviation variant for 'Bridget' — encountered in two records this batch."
}
```

**Field notes:**

- `assessed_score` — Claude's adjusted score for the linkage, reflecting domain reasoning beyond the Splink score. May be higher or lower than the proposal score. Null for `new_person` and `new_place` decisions.
- `reasoning` — mandatory for all decisions. This is the audit trail. It must be specific enough that a researcher reviewing it later can understand why the decision was made.
- `relationship_inferences` — new Relationship assertions derived from role pairs in the record. Python commits these alongside the linkage.
- `variant_additions` — name variant pairs Claude recommends adding to the `name_variant` table.
- `session_notes` — optional batch-level observations. Cross-batch patterns, recommended variant additions, anything that is not specific to a single proposal.

### 3.8 Reasoning standards

Claude's reasoning is the primary audit trail for decisions made without researcher verification. The following standards apply:

**Be specific about the evidence.** "Name match strong" is not sufficient. "Surname Mulligan exact match; forename Patrick exact match after normalisation" is the standard.

**Name the constraint when it applies.** "GC03 census age drift: delta 3 years, within ±3 tolerance" is better than "age is close enough."

**State the conflict when one exists.** If accepting a proposal despite a feature mismatch, name the mismatch and explain why it does not disqualify the linkage. Do not silently accept.

**Distinguish rejection reasons.** A rejection because the birth year arithmetic is impossible (Patrick Mulligan b. ~1820 cannot be age 8 in 1901) is different from a rejection because a better candidate exists. Both are valid reasons; they are not the same reason.

**Flag genuinely ambiguous cases.** If two existing Person conclusions are both plausible matches and the features do not clearly distinguish them, that is a flag, not an accept. The researcher is better positioned to resolve genuine ambiguity than Claude is to guess.

### 3.9 What Claude does not do in an update knowledge session

- Claude does not query the database directly. Python provides all required context in the session packet.
- Claude does not create Event conclusions. Event linkage is handled by Python after person linkage is committed.
- Claude does not modify `raw_text` or any evidence-layer field.
- Claude does not accept a linkage solely on the basis of a high Splink score. The score is a prior; Claude's reasoning is the update.
- Claude does not reject a linkage solely on the basis of a low Splink score. A low score with strong co-occurring person coherence may be a sound linkage.

### 3.10 Session close

After Claude returns the decision array, Python:

1. Commits all `accept` and `accept_verify` junction rows with `verified = 0`
2. Creates new Person or Place conclusions for `new_person` and `new_place` decisions
3. Commits relationship inferences from Claude's `relationship_inferences` arrays
4. Adds variant table entries from Claude's `variant_additions` recommendations
5. Adds `flag` decisions to the flags table
6. Updates the proposal queue (removes processed proposals)
7. If the queue is not empty, prepares the next batch and initiates a new update knowledge session

The knowledge base is updated incrementally as each session closes. A failure mid-session does not roll back previously committed batches.

## 4. Operating Principles

These principles apply across both session types and govern how Claude operates as a GRA collaborator.

### Evidence layer is inviolable

Claude never modifies `raw_text`. The verbatim ingest string is sacrosanct — it is the permanent record of exactly what was ingested. Claude provides parsed field values; Python stores them alongside the preserved `raw_text`.

### Conclusions are mutable; evidence is not

A linkage decision made today can be revised tomorrow. If new evidence contradicts an existing conclusion, the correct response is to flag the contradiction, not to modify the historical record. Claude frames its decisions as researcher-reviewable assertions, not permanent facts.

### Domain knowledge over score obedience

Claude's value in the update knowledge session is precisely that it can reason about cases where the score is misleading. A score of 0.71 with strong co-occurring person coherence and a plausible birth year delta may be a sounder linkage than a score of 0.82 with a suspicious age trajectory. Claude uses the score as context, not as instruction.

### Geographic grounding

All reasoning is grounded in the geographic scope of the session. A "Patrick Mulligan" in Tullynaught DED is being evaluated against the known community of Tullynaught DED — not against all Patrick Mulligans in the database. The scope constraint is a feature, not a limitation: it is what makes community-scale genealogy tractable.

### Audit trail discipline

Every accept and reject decision must have reasoning sufficient for a researcher to evaluate it independently. Claude is not the final authority — the researcher is. Claude's role is to make the researcher's review efficient by handling the routine cases with documented reasoning, and surfacing the genuinely hard cases as flags.

## 5. Session Packet Reference

### Ingest exception packet

| Field | Description |
|---|---|
| `session_type` | Always `"ingest"` |
| `source` | Source object with `source_id`, `title`, `type`, `column_schema` |
| `ingest_summary` | Row counts: attempted, committed, exceptions |
| `parse_exceptions` | Array of exception objects; one per unresolved row |

### Update knowledge packet

| Field | Description |
|---|---|
| `session_type` | Always `"update_knowledge"` |
| `scope` | Source, geographic scope string |
| `community_summary` | Conclusion counts for the geographic scope; record link status |
| `proposals` | Array of proposal objects; maximum 20 per session |
| `active_flags` | Unresolved flags from previous sessions in this scope; included for context |

### Proposal object

| Field | Description |
|---|---|
| `proposal_id` | Unique identifier for this proposal within the session |
| `record_id` | The Record being proposed for linkage |
| `candidate_type` | `"person"` or `"place"` |
| `candidate_id` | The conclusion object being proposed as the match; null for new_person/new_place candidates |
| `score` | Splink score after constraint adjustment |
| `score_version` | Algorithm version that produced the score |
| `features` | Feature dictionary: similarity scores, birth year delta, place match, co-person overlap |
| `record_summary` | Key fields from RecordedPerson and RecordedEvent for this record |
| `candidate_summary` | Key fields from the candidate conclusion: label, names, birth year est, supporting records, known relationships |
| `constraint_notes` | GC-coded notes from the constraint application pass |
